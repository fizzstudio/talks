4: Why visual?

  • visual channel: 10 million bits / second
  • 30–60% of brain dedicated to visual processing
    • fast visual response: "seeing dinner or being dinner"
  • touch: 8%, hearing: 2%
  • Detect, match, and make sense of patterns

8: Fast and slow

  • System 1: Fast, automatic, frequent, emotional, stereotypic, unconscious
    • pre-attentive attributes
    • gestalt principles
  • System 2: Slow, effortful, infrequent, logical, calculating, conscious
    • thinking about meaning
  • dataviz uses fast visual systems to convey data, then slow systems to allow user to explore it and think about it
  • slow system
    • cognitively expensive (mentally taxing)
    • … slow
  • goal is to reduce the speed and effort in accomplishing tasks in a non-visual way

9: Cognitive vs Perceptual Design Distinction

  • cognitive load: the amount of mental effort that we use to get the information that we need. A good data visualization minimizes the cognitive load while also communicating the message accurately. There are three types of cognitive load,
    • intrinsic: baseline cognitive load pertaining to how much brain power is needed to understand an idea. Different types of tasks will require more or less thought or attention.
    • extraneous: amount of extra brain power needed to perceive and understand a topic
    • germane: a way for the brain to look for patterns to develop context. As with extraneous load, it is to help take a cognitive issue and present it in an easy and meaningful way. We'll use these three types of cognitive loads instinctively throughout this specialization.

10: Memories

  • Sensory: hyper-short-term , automatic
    • iconic memory: visual, 100ms duration, light trails (gone in a flash)
    • haptic memory: touch, ~2s duration, only recently identified, not much is known
    • echoic memory: auditory, 3-4s duration, high-density
  • Short-term: information encoding and storing
  • Long-term: more about recall, less immediately relevant to dataviz, but visual does help with retention (would other sensory encodings help, too? further study)

11: Affordance and Equivalence

  • Long lost Jane Austin novel
  • Each feature of the chart includes an affordance and a set of tasks that it is geared toward
    • Affordances
    • For each affordance or task, we need to isolate the channel and mechanism and find accessible equivalent. Maybe iconic memory can be replaced with echoic or haptic?
    • Reduce cognitive load as much as possible
    • Anticipate user's needs
    • Tasks are typically implicit: comparison with other data point, position in trend, assessment with regard to whole, categorizing (filtering/sorting), navigation/focus
    • For each task, we need to make it explicit

12: Dashboards

  • multiple charts on the same page
  • usually interrelations between charts
  • priorities:
    • let the user know there are multiple charts (easy!)
    • describe the purpose of each chart (may be manual)
    • describe how the charts relate to each other (may be manual)
    • Good news! You already made these decisions when you designed the dashboard, so you can just write a short description, and that will benefit everyone

13: One size does not fit all

  • In general, you should make sure that your dataviz is perceptible to all
  • But knowing your audience is important
    • Different audiences may have more or less expertise in a topic, or have different goals
    • Some differences may be cognitive ability
    • Create user personas
  • Consider your target audience, and create your dataviz with them as your primary focus

14: Deep Graphics

15: Free pass

  • Until recently, most data visualizations were not called out in accessibility assessments
  • "Write alt text or long description"
  • "Create a hidden data table"
  • "You can't make dashboards accessible": "I’ll begin by stating my fundamental position: a dashboard that is accessible to the blind is a contradiction in terms." - Stephen Few https://www.perceptualedge.com/blog/?p=1756

16: A table is not equivalent access

  • increased burden on the user
  • doesn't explicitly show many relationships in data
  • if a table were just as accessible, we wouldn't use data viz
  • Landscape is changing, and it will be increasingly hard to get a free pass

17: Pretty persuasions

  • Find the right way to convince your organization to be early adopters of accessible dataviz
  • Credibility and ethics are important to data scientists. Get the data scientist on your side.
  • Doesn't cost any more than inaccessible charts, and no more difficult; dataviz already generated by library, just use accessible one
  • Embedding data in your visualization is better for users for
    • reuse
    • SEO
    • smart speakers and voice services
    • restyling / rebranding
    • future proofing

18: Levels of accessibility in dataviz

  • 0: None (benefit: can be automated)
  • 1. Provide generic alt-txt: "barchart", "pie chart of market segment size"
  • 2. Provide detailed summary in alt-txt: "Line chart showing market trends, with a downward trend over the last year", "bar chart with 12 bars; the highest value is 1500"
  • 3. Provide hidden content (e.g. a table) with the same information
  • 4. Serial access to data: table alternative
  • 5. Automatically generic task-oriented contextual options

19: User interface for data

  • Different types of data visualizations are optimized for performing different tasks
    • comparison to whole
    • comparison to other datapoints
    • trend over time
    • breakdown of categories
    • connection between items
  • Break down interface into sets of tasks oriented around optimizations

20: Libraries and tools:

  • Most professional packages let you control colors, so you can make your own accessibility
  • Highcharts
    • good range of charts and graphs
    • serial navigation of each data point
    • not turned on by default
    • fairly pricey
    • developer-oriented
    • visible source, but not open source (commercial license)
  • SAS
    • analytics and programming language, complex, full-featured
    • free accessible chart authoring tool for schools
    • interface heavily geared toward sonification as first pass for noting trends
    • very expensive
    • proprietary
  • Semiotic
    • good range of charts and graphs
    • rudimentary accessibility, but growing
    • serial navigation of each data point
    • developer-oriented
    • free
    • open source
  • Fizz Charts
    • alpha release
    • accessible authoring
    • UI-driven, with developer API
    • accessible by default
    • Fizz Reader provides task-oriented interface
    • hoping to make open source!
